- Got minimal Gluon kernel to compile with `mfma` (non-scaled)

**Attempt 2: mfma_scaled with e4m3 format**

# 2026-06-07 — MXFP8 Backward Kernel Optimization (Gluon Deep-Dive)

## Session: Gluon investigation for wgrad kernel

### Background
- Forward kernel geomean: 1533 TFLOPS (8 DSv3 shapes)
- Original backward kernel geomean: 759 TFLOPS (Triton 3.7.0, BLOCK_M=64)
- Target: ≥0.9× forward ≈ 1380 TFLOPS
- Forward-as-backward (v2): 0.96× kernel-only, 0.71× with data copies

### Gluon Investigation

Shekhar suggested using Gluon (low-level Triton API) since it should support whatever Triton supports.

**Attempt 1: Fixing wgrad_gluon.py syntax bugs**
- `instr_shape` needs 3 elements `[16, 16, 32]`, not 2
- MFMA operands must go through shared memory (DotOperandLayout + smem.load)
- `gl.constexpr` annotations required on all layout variables
- Got minimal Gluon kernel to compile with `mfma` (non-scaled)

**Attempt 2: mfma_scaled with e4m3 format**
- Hit definitive LLVM error on both Triton 3.7 and 3.6:
```
error: no matching matrix core intrinsic for mfma version 4
with instruction shape [16, 16, 32] and element types
A='f8E4M3FN', B='f8E4M3FN'.
```
- `mfma_scaled` + `e4m3` is NOT legalized for gfx950 in either version
- Forward kernel uses `tt.dot_scaled` which goes through standard Triton lowering (different codegen path)
- Gluon's raw `mfma_scaled` path doesn't have the type registered

**Attempt 3: Triton 3.6 downgrade**
- Installed Triton 3.6.0 to test if older version handles it
- 3.6 doesn't crash on BLOCK_M>64 but generates CATASTROPHICALLY slow code
- Backward geomean dropped from 759 → 184 TFLOPS
- Some shapes ran at ~42-85 TFLOPS with BLOCK_M=256
- `mfma_scaled` legalization error identical to 3.7
- Reverted to Triton 3.7.0

**Attempt 4: rocm/pytorch-nightly container**
- Pulled `rocm/pytorch-nightly:latest` (2026-04-29 image)
- Contains Triton 3.7.0 + PyTorch 2.13.0a0, NOT Triton 3.6 as expected
- Standard backward kernel: M64=565 TFLOPS, M128=365, M256=30 (identical degradation)
- `mfma_scaled` + `f8E4M3FN`: same legalization error
- Tested across ROCm 7.0 (pytorch-private), 7.1 (system), 7.2 (sgl-dev) — all identical
- Container/ROCm version doesn't fix the codegen issue

**Attempt 5: Gluon mfma (non-scaled) + manual per-32-element scaling**
- Approach: iterate M in chunks of 32, do `mfma` call per chunk, apply scale
- Layout mismatch bug: scales from buffer_load have BlockedLayout, incompatible with AMDMFMALayout for broadcast. Fixed by routing scales through shared memory with MFMA-compatible slice layout.
- **COMPILES AND RUNS**: 254 TFLOPS with BLOCK_M=32, BLOCK_N=128, BLOCK_K=128, NUM_WARPS=4
- **BLOCK_M>32 can't work**: `mfma` computes full reduction over all BLOCK_M elements — per-32-chunk scales can't be factored out post-hoc
- BLOCK_M=32 is too small: 256 reduction iterations per group (vs 64 for BLOCK_M=128), 4× MFMA calls + per-iteration scale overhead
- Inherently limited by the lack of `mfma_scaled` hardware support

### Complete Path Summary

| Approach | TFLOPS | × Forward | Status |
|----------|--------|-----------|--------|
| Forward-as-backward (kernel only) | 1467 | 0.96× | Viable, needs data layout work |
| Forward-as-backward (with copies) | 1083 | 0.71× | Acceptable fallback |
| Standard backward (BLOCK_M=64) | 759 | 0.50× | Current baseline |
| Gluon mfma + manual scales | 254 | 0.17× | Too many iterations |
| Gluon mfma_scaled + f8E4M3FN | ❌ | — | Missing LLVM intrinsic |
| Standard Triton M>64 | 42-372 | 0.03-0.24× | AMD codegen bug |

## MAJOR UPDATE: instr_shape=[32,32,64] IS legalized for mfma_scaled + f8E4M3FN on gfx950

### The Discovery
Previous conclusion ("mfma_scaled + f8E4M3FN not legalized") was **wrong** — it was using the wrong `instr_shape`. `[16,16,32]` fails, but **`[32,32,64]` works**. This matches the MFMA instruction shape used by the forward kernel's `tt.dot_scaled`.

### Gluon wgrad with [32,32,64]
- **1169 TFLOPS at BLOCK_M=128** (0.76× forward, 1.54× baseline of 759) — with None scales
- BLOCK_M=64 with [32,32,64]: 904 TFLOPS
- `mfma` (non-scaled) `[16,16,32]` with fp8 also compiles (reduces 32 elements = MXFP8 scale granularity)

### Two Blocker Bugs for Real Scale Tensors
1. **Shape bug**: Gluon's `_get_scale_shape` swaps B-scale dims to `[K//32, N]` but `verify_scaled_shape` expects `[N, K//32]`
2. **Layout bug** (hard blocker): `buffer_load` produces `BlockedLayout` but `dot_scaled` lowering expects `LinearLayout`. No conversion pass in Gluon pipeline. None scales auto-generate correct LinearLayout.

### Path Rankings (updated)

| Approach | TFLOPS | × Forward | Status |
|----------|--------|-----------|--------|
| Forward-as-backward (kernel only) | 1467 | 0.96× | ✅ Viable |
| Forward-as-backward (with copies) | 1083 | 0.71× | ✅ Fallback |
| Gluon [32,32,64] None scales | 1169 | 0.76× | ⚠️ No real MXFP8 scales |
| Gluon [32,32,64] real scales | — | — | ❌ Blocked by 2 bugs |
| Standard backward (BLOCK_M=64) | 759 | 0.50× | Baseline |

### Bug Reproducers
- `~/shekhar/grouped-gemms/repro_final.py` — failing case (mfma_scaled + real scales)
- `~/shekhar/grouped-gemms/repro_working.py` — working reference (None scales)

### Container Testing
Tested across `rocm/pytorch-nightly:latest`, `rocm/pytorch-private:2026-04-25`, `rocm/sgl-dev:2026-04-08` — all produce identical `mfma_scaled` behavior (only difference is which instr_shapes are legalized; [32,32,64] consistently works).

### Forward-as-Backward Remains King
0.957× forward kernel-only. Data copy overhead (per-group transposes within MoE loop) drops it to 0.71×. Optimizing the data pipeline (pre-batched layout) could close the gap.

### Root Cause Analysis
