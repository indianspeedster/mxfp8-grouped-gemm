- Caveat: PyTorch aggregate profiler API on that ROCm build reported `0.0 ms` device time, so the text kernel breakdown files were not useful for actual GPU durations.
- Useful performance signal from streamed large traces: hot inclusive CPU-profiler regions across sampled ranks 0/1/4/6 were `torchao::triton_to_mxfp8_dim0` (~463-624s), `PythonDispatchMode` (~469-478s), `aten::_to_copy` (~322-492s), `_MXFP8GroupedMM` (~232-312s), `aten::copy_` / `aten::to` (~165-246s), and `aten::zero_` / `aten::fill_` (~87-123s). These are nested/inclusive and should not be summed.
- Working hypothesis: 16B runtime is dominated by repeated MXFP8 conversion/copy overhead around grouped MM, especially `torchao::triton_to_mxfp8_dim0` and `_to_copy`, rather than obvious grouped-matmul compute. Next useful direction: cache/prepack MXFP8 weights or otherwise avoid repeated conversion/copy in the training step.

## 2026-06-09 - DeepSeek V3 BF16 vs MXFP8 follow-up

- Continued Torchtitan/DeepSeek V3 16B profiling on 8x MI355X in `/home/amd/mxfp8-training-20260607-053821/torchtitan`.
- Fixed profiler scheduling for CUDA-only text breakdowns. Clean steady-state MXFP8 capture used `steps=2`, `profile_freq=2`, `profiler_warmup=0`, `profiler_active=1`, so step 1 was skipped and step 2 captured. Trace files: `outputs/profiling/traces/iteration_2/rank*_cuda_only_breakdown.txt`.
- Initial clean-but-unpatched MXFP8 aggregate looked terrible: total device time across ranks `161.3s`, mean `20.16s/rank`, max `30.94s`. Top kernels were `FillFunctor<int>` `82.6s` / `2.08M` launches, `ncclDevKernel_Generic_1` `56.1s`, and `to_mxfp8_dim0_kernel.kd` `22.6s` / `2.60M` launches. `_mxfp8_grouped_mm_kernel.kd` was only ~`2.5ms` total.
- Ran a matched BF16 profile with the same shape: DeepSeek V3 16B, 8x MI355X, `seq_len=1024`, local batch 1, step 2 capture. Files: `outputs/profiling/traces/bf16_match/iteration_2/rank*_cuda_only_breakdown.txt`. BF16 aggregate was `2.54s` total device time, mean `318ms/rank`, max `375ms`; dominated by NCCL (`1.22s`, 48%) and ROCm/Tensile GEMMs (`514ms`, 20.2%).
- Created a Notion page under AMD titled `DeepSeek V3 16B BF16 vs MXFP8 Kernel Comparison MI355X` with side-by-side kernel/category totals, exact kernel names, and launch/owning-function notes. Page URL given to Shekhar: `https://app.notion.com/p/DeepSeek-V3-16B-BF16-vs-MXFP8-Kernel-Comparison-MI355X-37a74178c99781fb945edd79d55f7c33`.
- Debugged why unpatched MXFP8 E2E looked bad. Exact source was Triton autotuning inside `torchao` MXFP8 quantization: `triton_to_mxfp8_dim0` -> `to_mxfp8_dim0_kernel` -> Triton autotuner `benchmark()` -> `triton.testing.do_bench()` -> AMD driver `clear_cache(cache)` -> `cache.zero_()` on a `(67108864,) int32` CUDA tensor (~256MB). One rank/step saw 468 `triton_to_mxfp8_dim0` Python calls but 2,918,395 `Tensor.zero_()` calls.
- Root cause: `to_mxfp8_dim0_kernel` has 32 Triton autotune configs and autotune key includes `n_rows`; MoE routed-token counts vary, causing repeated autotune sweeps for many dynamic shapes. The bad MXFP8 profile was mostly measuring Triton benchmarking cache flushes, not steady-state MXFP8 math.
- Validation patch clamped `to_mxfp8_dim0_kernel` to one config and removed/avoided the dynamic `n_rows` autotune churn. Step 1 improved from ~230s to ~16s on rank0 (`tps 4` -> `tps 60`). Patched step-2 profile became normal: total device time `2.492s` across ranks, mean `311.5ms/rank`, basically same as BF16 `317.7ms/rank`.
- Patched MXFP8 step-2 category readout: NCCL `46.1%`, copy/index/gather `20.2%`, other/optimizer `18.4%`, MXFP8 convert `3.1%`, MXFP8 grouped MM `2.7%`, MXFP8 wgrad `2.1%`, fill/zero `0.9%` (down from 51.2%).
- Important artifacts: bad count report `/tmp/mxfp8_count_report_rank0.txt`; zero stack log `/tmp/dsv3-debug-zero-stack.log`; single-config validation log `/tmp/dsv3-mxfp8-single-autotune-step1.log`; patched profiler log `/tmp/dsv3-mxfp8-single-autotune-profile.log`; patched trace files `/home/amd/mxfp8-training-20260607-053821/torchtitan/outputs/outputs/profiling/traces/mxfp8_single_autotune/iteration_2/rank*_cuda_only_breakdown.txt`.
- Recommended next fixes for Shekhar: patch torchao MXFP8 dim0 autotune for MoE use by using a fixed known-good config or coarser key that excludes routed `n_rows`; separately add expert weight prepacking/caching so `w1/w2/w3` are not re-quantized every forward/dgrad.
