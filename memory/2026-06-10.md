# 2026-06-10 — Gluon MXFP8 Forward GEMM Optimization

## Key Breakthrough: N-slicing Works
- Discovered double-counting bug: both `b_base = b_ptr + pid_n * BN * stride_bn` AND `offs_bnl = pid_n * BN + arange(HN)` added pid_n offset → 2x offset, causing OOB GPU faults at multi-tile grids
- Fix: remove pid_n from arange offsets, compute masks separately using absolute positions
- Same bug for A offsets (pid_m double-counted)

## Constraints Found
- **HN == BK required**: B shared memory layout constraint. BN=128 needs BK=64, BN=256 needs BK=128
- **M384 fails**: Exceeds SMEM limits (48KB A + 16KB B × 2 = 80KB for M384 N128 K64 with NB=2)
- **NW=8 fails**: Triton "Layout has 8 warps per CTA, but context requires 4 warps per CTA"
- **UR=2 slower**: 1062 vs 1234 TFLOPS (prefetch overhead)
- **3-buffer prefetch slower**: 666 vs 1228 TFLOPS (more SMEM pressure)

## Scale Investigation (Today's Session)
### E8M0 encoding bug discovered
- **0x38 ≠ 1.0**: 0x38 = 56 → 2^(56-127) = 2^-71 ≈ 8.5e-22 (essentially zero)
- **0x7F = 1.0**: 127-127=0 → 2^0 = 1.0 ✓
- Fix: pad dummy scales with 0x7F not 0x38

### K//32 vs KS stride bug discovered
- When problem K ≠ BLOCK_K, scale row stride must be `K//32` (full tensor stride), NOT `KS = BLOCK_K//32`
- Simple test (K=128, BK=128) worked by coincidence (128/32=4 both ways)
- Repro test with K=256, BK=128 exposed the bug
- Fix: use `stride_asm = a_scales.stride(0)` (=K//32) not constexpr KS

### Scale loading in dot_scaled
- `tl.dot_scaled(a8, a_sc, "e4m3", b8, ws_l, "e4m3")` compiles and runs with loaded uint8 scales
- B scale layout: `[HALF_N, KS]` works, `[KS, HALF_N]` fails (scale factor 2 error)
- Scale pointer setup must use full-tensor strides, not tile-local constexprs

### Real scales benchmark
- **With real E8M0 scales: 1501 TFLOPS geomean** (down from 1821 without scales)
- Drop from scale loading overhead (extra loads per iteration)
- **Numerical discrepancy**: max error ~103 vs dequantized reference — suggests Triton's dot_scaled interprets E8M0 differently than 2^(byte-127) OCP spec, or lowering issue in gfx950-tutorial fork
- Max error vs pure BF16 ref: 473 (expected for FP8 3-mantissa-bit quantization)

## Scale Correctness — Root Cause Identified (Sub-agent)
- Deep analysis of MFMA lowering in gfx950-tutorial Triton fork
- E8M0 `2^(byte-127)` confirmed correct in 3 code paths (AccelerateAMDMatmul.cpp, UpcastMXFPToLLVM.cpp, deduceScaleFactor)
- **Root cause: MFMA.cpp `getValuesFromDotOperandLayoutStruct` (line 462-467)** — when `numRepK=1`, layout packs multiple M/N rows' scales into single register with wrong byte selection
- **Confirmed by NaN output** on known-value test (`scale_correctness.py`)
- **SCALE_FINDINGS.md** written: 6-section report with code references, lowering walkthrough, 5 hypotheses
- Fix requires editing MFMA.cpp in Triton source + rebuild

## Wgrad Breakthrough
- Native backward kernel: **611-683 TFLOPS — SLOWER than BF16** (~1100 TFLOPS)
- Root cause: native wgrad loops over all M rows per (N,K) tile, few CTAs
- **Fix: repurpose fused forward kernel for wgrad**
  - Forward computes C = A @ B^T. Wgrad = GO @ IA^T
  - Map: A = stacked GO [E*N, Mg], B = stacked IA [E, K, Mg]
  - Forward output [E*N, K] = grad_W reshaped to [E, N, K]
- **Result: 1839 TFLOPS geomean, 1.67x vs BF16** 🔥
- Module: `wgrad_fused.py` with `wgrad_fused(go, ia, offs)` and `wgrad_prepare`/`wgrad_kernel` pattern
- Data stacking is O(E*N*Mg) copy — done once per backward pass, not per kernel call
